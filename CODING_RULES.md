# CODING_RULES.md

# app_hr 编码规范与 AI 开发约束

Version: 1.0

## 一、目标

本文档规定项目的编码规范、目录规范、AI（Codex/OpenCode
等）开发约束，所有新增代码均应遵守。

------------------------------------------------------------------------

## 二、总体原则

-   简单优于复杂。
-   可读性优先于技巧。
-   模块低耦合、高内聚。
-   单一职责原则（SRP）。
-   尽量避免重复代码（DRY）。

------------------------------------------------------------------------

## 三、目录规范

    app_hr.py
    modules/
    utils/
    config/
    templates/
    fonts/
    tests/

业务逻辑放入 `modules/`，工具函数放入 `utils/`。

------------------------------------------------------------------------

## 四、Python 编码规范

-   遵循 PEP 8。
-   使用 Python 3.12+。
-   全部使用 UTF-8 编码。
-   新代码必须添加类型注解。
-   公共函数必须有 docstring。

示例：

``` python
def load_excel(path: str) -> pd.DataFrame:
    \"\"\"读取 Excel 并返回 DataFrame。\"\"\"
```

------------------------------------------------------------------------

## 五、异常处理

禁止：

``` python
except:
    pass
```

应使用：

``` python
except Exception as e:
    logger.exception(e)
    raise
```

向用户展示友好的错误信息。

------------------------------------------------------------------------

## 六、日志规范

统一使用 logging。

记录：

-   文件读取
-   LLM 调用
-   Prompt 长度
-   执行耗时
-   异常

禁止打印调试信息到页面。

------------------------------------------------------------------------

## 七、Streamlit 规范

-   页面负责展示。
-   业务逻辑放入 modules。
-   不在页面直接写复杂分析代码。
-   不展示 AI 原始生成代码（调试模式除外）。

------------------------------------------------------------------------

## 八、LLM 调用规范

统一封装：

    modules/llm_client.py

不得在多个文件中重复创建 Client。

统一支持：

-   OpenAI
-   Qwen
-   GLM
-   DeepSeek
-   OpenAI Compatible API

------------------------------------------------------------------------

## 九、Prompt 管理

Prompt 独立维护。

推荐：

    prompts/

禁止把超长 Prompt 写死在业务代码中。

------------------------------------------------------------------------

## 十、代码执行规范

所有 AI 生成代码必须：

-   AST 校验
-   白名单校验
-   禁止危险模块
-   禁止文件写入
-   禁止网络访问

------------------------------------------------------------------------

## 十一、命名规范

类：

    ExcelLoader
    PromptBuilder
    SafeExecutor

函数：

    snake_case

常量：

    UPPER_CASE

------------------------------------------------------------------------

## 十二、配置规范

统一放入：

    config/

不得在代码中硬编码：

-   API Key
-   Base URL
-   模型名称
-   文件路径

------------------------------------------------------------------------

## 十三、测试规范

新增模块建议配套单元测试。

测试目录：

    tests/

------------------------------------------------------------------------

## 十四、Git Commit

建议：

-   feat:
-   fix:
-   refactor:
-   docs:
-   test:
-   chore:

一次 Commit 只完成一个逻辑目标。

------------------------------------------------------------------------

## 十五、AI 开发约束

AI 开发时必须：

1.  优先修改已有模块。
2.  避免重复实现功能。
3.  不随意修改公共接口。
4.  新增模块需说明职责。
5.  保持向后兼容。
6.  生成代码必须可维护。
7.  未经要求不要引入大型第三方框架。
8.  优先标准库与成熟依赖。
